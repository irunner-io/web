EC2 runners

EC2 is iRunner's full virtual machine compute backend. Each job gets a dedicated EC2 instance with complete isolation, root access, and support for any workload including Docker builds, GPU tasks, and nested virtualization.

Overview

When a workflow job targets an EC2 pool, iRunner launches a fresh EC2 instance (or claims one from a warm pool), registers a GitHub Actions runner on it, and tears the instance down after the job completes. The instance is fully ephemeral — nothing persists between jobs.

Best for

Docker builds, GPU workloads, nested virtualization, large disk needs, custom kernels, and any job that requires full VM-level isolation.

Instance types

Any EC2 instance type is supported. Choose based on your workload requirements:

FamilyExampleUse case
General purposem7i.large, m7i.xlargeBalanced CPU/memory for most CI jobs
Compute optimizedc7g.xlarge, c7i.2xlargeCompilation, test suites
Burstablet4g.medium, t3.largeLightweight jobs, cost-sensitive
GPUg5.xlarge, g5.2xlargeML training, GPU-accelerated builds
ARM64 / Gravitonc7g.large, m7g.xlargeARM builds, cost-efficient compute

Custom AMIs

Control the base image your runners use:

  • ami: "auto" — uses the latest iRunner-maintained Ubuntu 22.04 AMI (updated weekly)
  • ami: "ami-0abcdef1234567890" — use your own custom AMI with pre-installed tools
Tip

Custom AMIs dramatically reduce cold start time. Pre-bake Docker images, language runtimes, and large dependencies into your AMI to skip install steps.

Disk configuration

Set the root EBS volume size per pool with disk_size_gb. Defaults to 50 GB if not specified.

config.yaml (excerpt)
disk_size_gb: 100   # 100 GB gp3 root volume

Networking

EC2 runners launch in private subnets within your VPC. The control plane assigns instances to subnets from the pool configuration:

  • Instances run in private subnets with no public IP by default
  • Internet access is provided via NAT gateway
  • Security groups control inbound/outbound traffic (default: egress-only)
  • Instances communicate with the control plane over the internal ALB

Instance profile

Attach an IAM instance profile to grant runners AWS permissions (e.g., ECR pull, S3 access, Secrets Manager):

config.yaml (excerpt)
instance_profile: "arn:aws:iam::123456789012:instance-profile/ir-runner-profile"

Spot instances

Enable Spot for up to 90% cost savings on interruptible workloads:

config.yaml (excerpt)
spot_enabled: true

iRunner automatically falls back to on-demand if Spot capacity is unavailable.

Warm pools

Keep pre-launched instances ready for instant job pickup:

config.yaml (excerpt)
min_warm: 2   # always keep 2 instances warm and registered

Warm pool instances are pre-registered as GitHub runners and can pick up jobs in under 1 second. The control plane maintains the target count by launching replacements as warm instances are consumed.

Packages

Install packages on first boot using the packages list. These are installed via apt-get before the runner starts:

config.yaml (excerpt)
packages:
  - docker.io
  - python3
  - python3-pip
  - nodejs
  - npm
  - build-essential

Sudoers

Grant the runner user additional sudo permissions with sudoers_extra:

config.yaml (excerpt)
sudoers_extra: "runner ALL=(ALL) NOPASSWD: /usr/bin/apt-get, /usr/bin/systemctl"

Docker access

When docker.io is in the packages list, the runner user is automatically added to the docker group. This means workflows can run docker build, docker compose, and other Docker commands without sudo.

Note

The docker group is added after package installation during instance bootstrap. The runner process starts after group membership is applied, so no logout/login cycle is needed.

Full pool configuration example

config.yaml
pools:
  - name: ec2-large
    driver: ec2
    labels:
      - ir-ec2-large
      - ir-docker
    instance_type: m7i.large
    ami: "auto"
    disk_size_gb: 100
    spot_enabled: true
    min_warm: 2
    instance_profile: "arn:aws:iam::123456789012:instance-profile/ir-runner-profile"
    subnets:
      - subnet-0a1b2c3d4e5f6a7b8
      - subnet-9c8d7e6f5a4b3c2d1
    security_groups:
      - sg-0123456789abcdef0
    packages:
      - docker.io
      - python3
      - python3-pip
      - build-essential
      - jq
      - unzip
    sudoers_extra: "runner ALL=(ALL) NOPASSWD: ALL"

Workflow usage

Reference the pool label in your workflow's runs-on field:

.github/workflows/ci.yml
jobs:
  build:
    runs-on: ir-ec2-large
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t myapp .
      - run: docker compose up -d
      - run: make integration-test

Cold start times

ConfigurationCold start
No warm pool, default AMI45-90 seconds
No warm pool, custom AMI (pre-baked)30-60 seconds
Warm pool (min_warm > 0)< 1 second (instant)
Tip

For production workloads, set min_warm: 1 or higher to eliminate cold starts. Warm instances cost only the EC2 hourly rate while idle — typically $0.03-0.10/hour depending on instance type.
