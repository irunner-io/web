ECS Fargate

ECS Fargate is iRunner's serverless container compute backend. Jobs run in ephemeral Fargate tasks with no infrastructure to manage — no AMIs, no instance lifecycle, no patching. Just specify a container image and size.

Overview

When a workflow job targets an ECS pool, iRunner launches a Fargate task in your ECS cluster, runs the GitHub Actions runner inside the container, and tears the task down when the job completes. There are no servers to manage — AWS handles the underlying compute.

Best for

Fast lightweight jobs, CI test suites, linting, formatting, type checking, and any workload that doesn't require Docker-in-Docker or privileged access.

No AMI management

Unlike EC2, Fargate pools use container images instead of AMIs. You can use iRunner's maintained runner image or bring your own:

  • Default image includes Ubuntu 22.04, common build tools, Node.js, Python, and the GitHub Actions runner
  • Custom images let you pre-install your specific toolchain for faster job starts

ecs.ephemeral_storage_gbNoEphemeral storage in GB (default: 21, max: 200)

Fargate Spot

Enable Fargate Spot for up to 70% cost savings on fault-tolerant workloads:

config.yaml (excerpt)
ecs:
  spot_enabled: true
Note

Fargate Spot tasks can be interrupted with a 2-minute warning. iRunner will automatically retry interrupted jobs on on-demand capacity.

Dynamic sizing with label patterns

Use label patterns to let workflows request specific resource configurations without pre-defining a pool for every size:

config.yaml (excerpt)
pools:
  - name: ecs-dynamic
    driver: ecs
    label_pattern: "ir-ecs-{cpu}cpu-{mem}g"
    ecs:
      cluster_arn: "arn:aws:ecs:us-east-1:123456789012:cluster/ir-runners"
      # CPU and memory are derived from the label pattern

Workflows can then request exact resources:

.github/workflows/ci.yml
jobs:
  test:
    runs-on: ir-ecs-2cpu-4g    # 2 vCPU, 4 GB memory
  lint:
    runs-on: ir-ecs-1cpu-2g    # 1 vCPU, 2 GB memory

Full pool configuration example

config.yaml
pools:
  - name: ecs-small
    driver: ecs
    labels:
      - ir-ecs-small
      - ir-lint
    ecs:
      cluster_arn: "arn:aws:ecs:us-east-1:123456789012:cluster/ir-runners"
      image: "123456789012.dkr.ecr.us-east-1.amazonaws.com/ir-runner:latest"
      execution_role_arn: "arn:aws:iam::123456789012:role/ir-ecs-execution"
      task_role_arn: "arn:aws:iam::123456789012:role/ir-ecs-task"
      size: fargate.medium
      spot_enabled: true
      ephemeral_storage_gb: 40
      subnets:
        - subnet-0a1b2c3d4e5f6a7b8
        - subnet-9c8d7e6f5a4b3c2d1
      security_groups:
        - sg-0123456789abcdef0

Workflow usage

.github/workflows/ci.yml
jobs:
  lint:
    runs-on: ir-ecs-small
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck

  test:
    runs-on: ir-ecs-2cpu-4g
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test -- --coverage

Tip

Keep your custom container images small. A lean image (under 500 MB) starts significantly faster than a bloated one. Use multi-stage builds and only include runtime dependencies.

Limitations

  • No Docker-in-Docker — Fargate does not support privileged containers. Use EC2 pools for Docker build workloads.
  • Max 200 GB ephemeral storage — for larger disk needs, use EC2.
  • No GPU support — Fargate does not offer GPU instance types.
  • Max 16 vCPU / 32 GB per task — for larger compute, use EC2.
