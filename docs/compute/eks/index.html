EKS pods

EKS is iRunner's Kubernetes-native compute backend. Jobs run as ephemeral pods in your existing EKS cluster, leveraging your cluster autoscaler, node pools, and Kubernetes-native tooling for maximum flexibility.

Overview

When a workflow job targets an EKS pool, iRunner creates a pod in your cluster's designated namespace, runs the GitHub Actions runner inside it, and deletes the pod when the job completes. This integrates naturally with your existing Kubernetes infrastructure — node autoscaling, RBAC, network policies, and observability all work as expected.

Best for

Teams with existing Kubernetes clusters, fine-grained resource control, sidecar patterns, GPU/ARM node affinity, and environments where Kubernetes is the standard deployment platform.

Prerequisites

  • An existing EKS cluster with at least one node group
  • Cluster autoscaler (Karpenter or Cluster Autoscaler) for dynamic scaling
  • iRunner control plane has RBAC access to create/delete pods in the target namespace
  • A Kubernetes service account with appropriate IAM roles (IRSA)

Configuration fields

FieldRequiredDescription
driverYesMust be eks
eks.cluster_nameYesName of the EKS cluster
eks.namespaceNoKubernetes namespace for runner pods (default: ir-runners)
eks.service_accountNoService account for pods (default: ir-runner)
eks.imageNoContainer image for runner (defaults to iRunner's managed image)
eks.node_selectorNoNode selector labels for pod scheduling
eks.tolerationsNoTolerations for tainted nodes (GPU, ARM, spot, etc.)
eks.resources.cpuNoCPU request/limit (default: 2)
eks.resources.memoryNoMemory request/limit (default: 4Gi)

Cluster autoscaler integration

EKS pools rely on your cluster autoscaler to provision nodes when pod demand exceeds capacity. iRunner simply creates pods — scaling is handled by:

  • Karpenter — recommended. Launches right-sized nodes in seconds based on pod requirements.
  • Cluster Autoscaler — scales existing node groups based on pending pods.
Tip

With Karpenter, configure provisioners that match your runner workloads. A dedicated provisioner for CI pods ensures runners don't compete with production workloads for node capacity.

Pod security contexts

Runner pods run as non-root by default with a restricted security context:

Generated pod spec (simplified)
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  fsGroup: 1000
  seccompProfile:
    type: RuntimeDefault

This is compatible with most Pod Security Standards. If your workload needs elevated privileges (e.g., for Docker-in-Docker), use EC2 pools instead.

Node affinity for GPU or ARM

Use node_selector and tolerations to target specific node types:

config.yaml (GPU pool excerpt)
eks:
  node_selector:
    nvidia.com/gpu: "true"
    kubernetes.io/arch: amd64
  tolerations:
    - key: nvidia.com/gpu
      operator: Exists
      effect: NoSchedule
config.yaml (ARM64 pool excerpt)
eks:
  node_selector:
    kubernetes.io/arch: arm64
  tolerations:
    - key: arch
      value: arm64
      effect: NoSchedule

Full pool configuration example

config.yaml
pools:
  - name: eks-small
    driver: eks
    labels:
      - ir-eks-small
      - ir-k8s
    eks:
      cluster_name: "production-cluster"
      namespace: "ir-runners"
      service_account: "ir-runner"
      image: "123456789012.dkr.ecr.us-east-1.amazonaws.com/ir-runner:latest"
      resources:
        cpu: "2"
        memory: "4Gi"
      node_selector:
        workload-type: ci
      tolerations:
        - key: dedicated
          value: ci-runners
          effect: NoSchedule

Workflow usage

.github/workflows/ci.yml
jobs:
  test:
    runs-on: ir-eks-small
    steps:
      - uses: actions/checkout@v4
      - run: go test ./...

  build-arm:
    runs-on: ir-eks-arm64
    steps:
      - uses: actions/checkout@v4
      - run: make build-arm64

Cold start times

ScenarioCold start
Node available, image cached10-20 seconds
Node available, image pull required20-45 seconds
Node scale-up required (Karpenter)60-90 seconds
Node scale-up required (Cluster Autoscaler)90-180 seconds
Tip

For fast cold starts, keep at least one warm node in your CI node group and pre-pull runner images using a DaemonSet. This ensures pods schedule instantly without waiting for image pulls or node provisioning.

Sidecar patterns

EKS pools support sidecar containers for common CI patterns. Configure additional containers that run alongside the runner:

  • Database sidecars — PostgreSQL, MySQL, Redis for integration tests
  • Service mesh — Envoy/Istio proxies for service-to-service testing
  • Log collectors — Fluent Bit or custom log shippers

IRSA (IAM Roles for Service Accounts)

Grant runner pods AWS permissions via IAM Roles for Service Accounts. This is the recommended approach over instance profiles:

Service account annotation
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ir-runner
  namespace: ir-runners
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/ir-eks-runner-role

Limitations

  • No Docker-in-Docker by default — pods run as non-root. Use Docker-outside-of-Docker (bind mount) or Kaniko for image builds.
  • Cluster dependency — requires a healthy EKS cluster. Cluster issues affect runner availability.
  • Resource contention — runner pods share cluster resources. Use dedicated node groups or resource quotas to prevent contention with production workloads.
