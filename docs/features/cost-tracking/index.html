Per-Job Cost Tracking

Every job that runs on iRunner gets a cost estimate based on instance type, duration, and pricing model (spot vs on-demand). Costs are recorded in DynamoDB and emitted as OpenTelemetry metrics for dashboards and alerting.

How costs are calculated

When a job completes, the control plane calculates its cost using:

  1. Instance type hourly rate: looked up from a built-in pricing table (updated weekly)
  2. Duration: wall-clock time from runner start to runner termination (includes setup overhead)
  3. Pricing model: spot or on-demand rate applied based on how the instance was launched
  4. Additional costs: EBS storage (prorated), data transfer estimates

The result is an estimated cost in USD, stored alongside the job metadata.

DynamoDB jobs table

Every job is recorded in the DynamoDB jobs table with the following attributes:

AttributeTypeDescription
job_idString (PK)GitHub workflow job ID
repoStringRepository full name (org/repo)
workflowStringWorkflow filename
poolStringPool name that served the job
instance_typeStringEC2 instance type or ECS task size
spotBooleanWhether spot pricing was used
started_atNumberUnix timestamp of job start
completed_atNumberUnix timestamp of job completion
duration_secondsNumberTotal wall-clock duration
cost_usdNumberEstimated cost in USD
statusStringJob outcome (success, failure, cancelled)

Metrics

Cost metrics are emitted via OpenTelemetry and available in Prometheus/CloudWatch:

Key metric names
# Cost per job (histogram)
ir_job_cost_usd_bucket{pool, instance_type, spot, repo}
ir_job_cost_usd_sum{pool, instance_type, spot, repo}
ir_job_cost_usd_count{pool, instance_type, spot, repo}

# Duration per job (histogram)
ir_job_duration_seconds_bucket{pool, instance_type, spot, repo, status}
ir_job_duration_seconds_sum{pool, instance_type, spot, repo, status}
ir_job_duration_seconds_count{pool, instance_type, spot, repo, status}

# Total spend (counter, monotonically increasing)
ir_total_spend_usd{pool, spot}

# Spot savings (counter)
ir_spot_savings_usd{pool}

# Jobs completed (counter)
ir_jobs_completed_total{pool, status, spot}

Querying costs

Via the internal API

Terminal
# Get cost for a specific job
curl -s http://control-plane.ir.internal:8080/internal/jobs/12345678 | jq .cost_usd

# Get total spend for a repository in the last 7 days
curl -s "http://control-plane.ir.internal:8080/internal/costs?repo=myorg/myrepo&days=7" | jq .

# Get spend breakdown by pool
curl -s "http://control-plane.ir.internal:8080/internal/costs/by-pool?days=30" | jq .

Via CloudWatch Metrics

All metrics are published to the iRunner CloudWatch namespace with dimensions for pool, repository, and pricing model. You can create CloudWatch dashboards and alarms directly.

Via Prometheus/Grafana

Example PromQL queries for cost dashboards:

PromQL examples
# Daily spend by pool (last 24h)
increase(ir_total_spend_usd[24h])

# Average cost per job by repository
ir_job_cost_usd_sum / ir_job_cost_usd_count

# Spot savings percentage
ir_spot_savings_usd / (ir_total_spend_usd + ir_spot_savings_usd) * 100

# Top 5 most expensive repositories (last 7d)
topk(5, increase(ir_total_spend_usd{repo!=""}[7d]))

Cost allocation

Use the repo dimension to allocate CI costs back to teams or projects. Common patterns:

  • Per-repository: Direct cost attribution based on repo name
  • Per-team: Group repositories by team using Grafana label joins or external mapping
  • Per-workflow: Track costs by workflow name to identify expensive pipelines
Tip

Set up a CloudWatch alarm on ir_total_spend_usd to get notified when daily CI spend exceeds your budget threshold. This catches runaway workflows or misconfigured matrix builds early.
