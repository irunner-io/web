Custom Images

Bring your own AMI for EC2 runners or your own container image for ECS/EKS. Pre-bake your toolchain, dependencies, and configuration for faster builds and consistent environments.

Why custom images?

The default iRunner images include a minimal set of tools (git, Docker, the GitHub Actions runner). For most teams, additional tools are needed: specific language runtimes, build tools, system libraries, or proprietary software. Custom images solve this by pre-installing everything at image build time.

  • Faster builds: No apt-get install or npm install -g at runtime
  • Consistent environments: Every job gets the exact same toolchain version
  • Reduced network traffic: Dependencies are baked in, not downloaded every run
  • Compliance: Pre-approved images with security patches applied

EC2 custom AMI

For EC2-backed pools, specify your AMI ID in the pool configuration:

config.yaml
pools:
  - name: linux-custom
    platform: ec2
    instance_type: m7i.2xlarge
    ami: ami-0your-custom-ami-id
    disk_size_gb: 100
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
    security_group_ids:
      - sg-runners-001

  - name: linux-golang
    platform: ec2
    instance_type: c7i.xlarge
    ami: ami-0golang-prebaked
    disk_size_gb: 50
    subnet_ids:
      - subnet-aaa111
    security_group_ids:
      - sg-runners-001

label_mappings:
  ir-custom: linux-custom
  ir-golang: linux-golang

AMI requirements

Your custom AMI must include:

RequirementDetails
Operating systemUbuntu 22.04 or 24.04 (Amazon Linux 2023 also supported)
GitHub Actions runnerPre-installed at /opt/actions-runner/
DockerInstalled and the runner user added to the docker group
iRunner agentThe bootstrap script at /opt/ir/bootstrap.sh
Cloud-initFor receiving user-data at launch
Tip

Start from iRunner's base AMI and add your customizations on top. This ensures all required components are present. Build your AMI with Packer for reproducibility.

Example Packer template

packer/custom-runner.pkr.hcl
source "amazon-ebs" "runner" {
  ami_name      = "ir-custom-runner-{{timestamp}}"
  instance_type = "m7i.large"
  region        = "us-west-2"
  source_ami    = "ami-0irunner-base"   # iRunner's published base AMI

  ssh_username = "ubuntu"
}

build {
  sources = ["source.amazon-ebs.runner"]

  # Install your tools
  provisioner "shell" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y golang-1.22 nodejs npm python3-pip",
      "sudo npm install -g pnpm@9",
      "pip3 install awscli boto3",
    ]
  }

  # Copy configuration files
  provisioner "file" {
    source      = "configs/"
    destination = "/opt/ir/configs/"
  }
}

ECS/EKS custom container image

For ECS Fargate and EKS pools, specify a container image URI:

config.yaml
pools:
  - name: fargate-node
    platform: ecs
    cpu: 4096
    memory: 8192
    image: "123456789012.dkr.ecr.us-west-2.amazonaws.com/runner-node:20-latest"

  - name: fargate-python
    platform: ecs
    cpu: 2048
    memory: 4096
    image: "123456789012.dkr.ecr.us-west-2.amazonaws.com/runner-python:3.12"

  - name: eks-java
    platform: eks
    cpu_request: "4"
    memory_request: "8Gi"
    image: "123456789012.dkr.ecr.us-west-2.amazonaws.com/runner-java:21"
    namespace: runners
    service_account: runner-sa

label_mappings:
  ir-node: fargate-node
  ir-python: fargate-python
  ir-java: eks-java

Container image requirements

Your container image must include:

RequirementDetails
BaseUbuntu 22.04+ or Debian 12+
GitHub Actions runnerInstalled at /opt/actions-runner/
EntrypointThe iRunner entrypoint script (/opt/ir/entrypoint.sh)
UserNon-root runner user (UID 1000)

Example Dockerfile

Dockerfile
FROM 123456789012.dkr.ecr.us-west-2.amazonaws.com/runner-base:latest

# Install Node.js 20
RUN curl -fsSL https://deb.nodesource.com/setup_20.x | bash - \
    && apt-get install -y nodejs \
    && npm install -g pnpm@9

# Install additional tools
RUN apt-get install -y \
    jq \
    yq \
    gh

# Pre-cache common dependencies (optional)
COPY package.json /tmp/deps/
RUN cd /tmp/deps && npm install && rm -rf /tmp/deps

USER runner
WORKDIR /home/runner

Image update strategy

When you update your custom image:

  • EC2: Update the ami field in config.yaml and upload the new config. New runners will use the new AMI. Existing warm pool runners will be replaced on their next refresh cycle.
  • ECS/EKS: Push a new image to the same tag (e.g., :latest) or update the image field to a new tag. New tasks/pods will pull the updated image.
Note

When using mutable tags like :latest for ECS images, new tasks always pull the latest version. For EC2, you must update the AMI ID in config.yaml since AMI IDs are immutable.
