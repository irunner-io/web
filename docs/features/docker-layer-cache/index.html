Docker Layer Cache

iRunner provides cross-job Docker layer persistence backed by ECR and S3. Docker builds reuse cached layers from previous runs, dramatically reducing image build times on ephemeral runners.

The problem with ephemeral Docker builds

Since iRunner uses ephemeral runners (destroyed after each job), the local Docker layer cache is lost between builds. Without cross-job caching, every Docker build downloads and rebuilds all layers from scratch. For large images, this can add 5-15 minutes to every build.

How it works

iRunner's Docker layer cache uses a combination of ECR (for registry-based caching) and S3 (for blob storage):

  1. Docker BuildKit exports cache metadata to an ECR registry in your account
  2. Layer blobs are stored in a dedicated S3 bucket for efficient retrieval
  3. On the next build, BuildKit pulls cache manifests from ECR and retrieves unchanged layers from S3
  4. Only modified layers are rebuilt and re-uploaded

Configuration

config.yaml
docker_cache:
  enabled: true
  docker_registry: "123456789012.dkr.ecr.us-west-2.amazonaws.com"
  docker_bucket: ir-west-docker-layers-123456789012
  region: us-west-2
  ttl_days: 7              # registry layers older than this are cleaned up
  max_cache_size_gb: 100   # per-repository cache size limit

FieldDescriptionDefault

Workflow usage

Use Docker Buildx with the registry cache backend. The runner is pre-authenticated to the ECR registry:

.github/workflows/docker-build.yml
jobs:
  build:
    runs-on: ir-large
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: 123456789012.dkr.ecr.us-west-2.amazonaws.com/myapp:${{ github.sha }}
          cache-from: type=registry,ref=123456789012.dkr.ecr.us-west-2.amazonaws.com/myapp:buildcache
          cache-to: type=registry,ref=123456789012.dkr.ecr.us-west-2.amazonaws.com/myapp:buildcache,mode=max

Using S3 cache backend directly

For larger caches, you can use the S3 cache backend with BuildKit:

.github/workflows/docker-build-s3.yml
jobs:
  build:
    runs-on: ir-large
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build with S3 cache
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: 123456789012.dkr.ecr.us-west-2.amazonaws.com/myapp:${{ github.sha }}
          cache-from: type=s3,region=us-west-2,bucket=ir-west-docker-layers-123456789012,name=myapp
          cache-to: type=s3,region=us-west-2,bucket=ir-west-docker-layers-123456789012,name=myapp,mode=max

Layer eviction

Cache layers are evicted based on two policies:

  • TTL-based: Layers not referenced within ttl_days are deleted via ECR lifecycle policies and S3 lifecycle rules
  • Size-based: When a repository's cache exceeds max_cache_size_gb, the oldest layers are evicted first
Note

The ECR registry used for Docker layer caching is separate from your application image registries. iRunner creates cache-specific repositories with the prefix ir-cache/ to avoid conflicts with your production images.

Performance impact

Typical improvements with Docker layer caching enabled:

ScenarioWithout cacheWith cacheImprovement
Node.js app (large node_modules)8 min1.5 min~80%
Go binary (multi-stage)5 min45 sec~85%
Python ML image (large deps)12 min2 min~83%

Prerequisites

  • ECR repository created in your account (the CloudFormation stack creates this automatically)
  • Runner IAM role has ecr:GetAuthorizationToken, ecr:BatchGetImage, ecr:PutImage permissions
  • S3 bucket for layer blobs (created by the CloudFormation stack)
