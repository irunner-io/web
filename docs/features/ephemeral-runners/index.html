Ephemeral Runners

Every iRunner runner is destroyed after completing a single job. This ephemeral model eliminates persistent state between jobs, ensuring a clean environment every time and reducing the attack surface of your CI infrastructure.

How it works

iRunner registers every runner with the --ephemeral flag when connecting to GitHub. This tells GitHub that the runner will only accept one job and then de-register itself. The lifecycle is:

  1. Provision: A new VM, container, or pod is created for the job
  2. Register: The runner registers with GitHub using a just-in-time (JIT) token with --ephemeral
  3. Execute: The runner picks up exactly one job and runs it
  4. Terminate: After the job completes (success or failure), the runner de-registers and the compute resource is destroyed
  5. Cleanup: The EC2 instance is terminated, ECS task is stopped, or K8s pod is deleted

Security benefits

Ephemeral runners provide significant security advantages over persistent self-hosted runners:

ThreatPersistent runnerEphemeral runner (iRunner)
Credential leakage between jobsSecrets from Job A accessible to Job BImpossible — environment destroyed
Malicious code persistenceMalware can survive between jobsDestroyed with the environment
Crypto minersCan run indefinitely in backgroundKilled when job completes
Lateral movementCompromised runner remains on networkEphemeral — no persistent foothold
Dirty state bugsPrevious job artifacts can affect next jobClean slate every time

Automatic cleanup

iRunner ensures runners are cleaned up in all scenarios:

  • Job success: Runner completes, de-registers, compute terminated immediately
  • Job failure: Same as success — compute terminated regardless of exit code
  • Job timeout: If a job exceeds its timeout (default: 6 hours), the control plane forcibly terminates the runner
  • Orphaned runners: The housekeeping loop detects runners that are registered but not running a job for too long, and terminates them
  • Spot interruption: AWS terminates the instance; the control plane detects the loss and updates state

Timeout configuration

config.yaml
runner:
  max_job_duration: 6h       # maximum time a single job can run
  idle_timeout: 30m          # max time a warm runner waits for a job
  orphan_check_interval: 60s # how often to check for orphaned runners

No persistent state

Because runners are ephemeral, there is no local state between jobs:

  • No local Docker layer cache (use Docker layer cache for cross-job caching)
  • No local file cache (use S3 cache instead)
  • No leftover processes or background services
  • No environment variables or shell history from previous jobs
  • No repository checkouts from other projects
Note

The ephemeral model means every job pays the cost of setup (checkout, dependency install, etc.). Use iRunner's S3 cache and custom images to minimize this overhead while maintaining the security benefits of ephemeral environments.

Comparison with persistent runners

AspectPersistent runnersEphemeral runners (iRunner)
SecurityShared state is a riskComplete isolation per job
MaintenanceRequires patching, monitoring driftImage-based, immutable
ScalingFixed fleet, manual scalingAutomatic, on-demand
Cost modelAlways-on (pay for idle)Pay per job (plus warm pool cost)
Local cacheAvailable (fast, but dirty)Not available (use S3 cache)
Setup timeInstant (already running)Cold-start unless warm pool configured

JIT runner tokens

iRunner uses GitHub's just-in-time runner registration API. Each runner gets a unique, short-lived registration token that is valid for exactly one registration. This means:

  • No long-lived runner tokens stored anywhere
  • Tokens cannot be reused if intercepted
  • Each runner has a unique identity for audit purposes
