Label Patterns

Label patterns enable dynamic resource allocation by letting developers specify compute requirements directly in the runs-on: label. Instead of pre-defining every possible combination, patterns use parameterized templates with constraints.

How it works

A label pattern is a template string with placeholders enclosed in curly braces. When a job requests a label, the control plane attempts to match it against configured patterns, extracts the parameter values, validates them against constraints, and provisions a runner with the resolved configuration.

This allows a single pattern to serve many different resource configurations without requiring individual pool definitions for each one.

Resolution flow

  1. Job requests runs-on: ir-ecs-2cpu-4g
  2. Control plane checks label_mappings first (no match)
  3. Control plane checks label_patterns — matches ir-ecs-{cpu}cpu-{mem}g
  4. Extracts: cpu=2, mem=4
  5. Validates against constraints: cpu must be in [1, 2, 4, 8, 16], mem must be in [2, 4, 8, 16, 32]
  6. Provisions an ECS task with 2 vCPU and 4 GB memory

Configuration

Label patterns are defined in the label_patterns section of config.yaml:

config.yaml
label_patterns:
  - pattern: "ir-ecs-{cpu}cpu-{mem}g"
    platform: ecs
    constraints:
      cpu: [1, 2, 4, 8, 16]
      mem: [2, 4, 8, 16, 32]
    mapping:
      cpu: "{cpu}048"      # converts vCPU count to Fargate CPU units
      memory: "{mem}096"   # converts GB to Fargate memory in MB (approximate)
    image: "123456789.dkr.ecr.us-west-2.amazonaws.com/runner:latest"
    spot_enabled: true

  - pattern: "ir-ec2-{size}"
    platform: ec2
    constraints:
      size: [small, medium, large, xlarge, 2xlarge]
    mapping:
      instance_type:
        small: m7i.large
        medium: m7i.xlarge
        large: m7i.2xlarge
        xlarge: m7i.4xlarge
        2xlarge: m7i.8xlarge
    ami: ami-0abc123def456
    disk_size_gb: 100
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
    security_group_ids:
      - sg-runners-001

  - pattern: "ir-eks-{cpu}cpu-{mem}g"
    platform: eks
    constraints:
      cpu: [1, 2, 4, 8]
      mem: [2, 4, 8, 16, 32]
    mapping:
      cpu_request: "{cpu}"
      memory_request: "{mem}Gi"
    namespace: runners
    service_account: runner-sa

Using patterns in workflows

Developers simply use the parameterized label in runs-on::

.github/workflows/ci.yml
jobs:
  lint:
    runs-on: ir-ecs-1cpu-2g    # minimal resources for linting
    steps:
      - uses: actions/checkout@v4
      - run: make lint

  test:
    runs-on: ir-ecs-4cpu-8g    # more resources for tests
    steps:
      - uses: actions/checkout@v4
      - run: make test

  build-image:
    runs-on: ir-ec2-large      # EC2 with mapped instance type
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t myapp .

  heavy-compile:
    runs-on: ir-eks-8cpu-32g   # large Kubernetes pod
    steps:
      - uses: actions/checkout@v4
      - run: make build-all

Constraints

Constraints prevent invalid or excessively large resource requests. If a requested value is not in the allowed list, the job is rejected with an error.

Constraint typeDescriptionExample
Enumerated valuesValue must be one of the listed options[1, 2, 4, 8, 16]
String enumsNamed sizes or tiers[small, medium, large]
Note

Label patterns are evaluated after static label_mappings. If a label matches both a static mapping and a pattern, the static mapping takes priority.

ECS CPU and memory combinations

When using ECS Fargate, be aware that not all CPU/memory combinations are valid. Fargate enforces specific pairings:

CPU (vCPU)Valid memory (GB)
0.250.5, 1, 2
0.51, 2, 3, 4
12, 3, 4, 5, 6, 7, 8
24, 5, 6, ... 16
48, 9, 10, ... 30
816, 17, 18, ... 60
1632, 33, 34, ... 120

Configure your constraints to only allow valid Fargate-compatible combinations.

Best practices

  • Use constraints to enforce cost guardrails (e.g., cap at 8 vCPU for most workloads)
  • Combine patterns with spot_enabled: true for cost-efficient dynamic allocation
  • Document available patterns for your team so developers know what labels to use
  • Start with a few common sizes and add more as needed
