Multi-Platform Support

iRunner supports GitHub Actions, GitLab CI, and Harness CI from a single deployment. The same compute pools serve all platforms — platform adapters handle the protocol translation between each CI system and iRunner's orchestration layer.

Supported platforms

PlatformStatusIntegration method
GitHub ActionsPrimary (GA)Webhook + GitHub App
GitLab CIAdapterGitLab runner protocol translation
Harness CIAdapterDelegate integration (port 3000)

Architecture

Each platform has an adapter that translates its native protocol into iRunner's internal job format:

Platform adapter flow
GitHub Actions:
  GitHub webhook → GitHub Adapter → Orchestrator → Pool → Runner

GitLab CI:
  GitLab Runner API ← GitLab Adapter → Orchestrator → Pool → Runner

Harness CI:
  Harness Delegate → port 3000 → Harness Adapter → Orchestrator → Pool → Runner

The orchestrator and pool layer are shared — a single pool can serve jobs from any platform.

GitHub Actions (primary)

GitHub Actions is the primary platform and uses the standard webhook-based flow:

  1. GitHub sends a workflow_job webhook when a job is queued
  2. The GitHub adapter validates the webhook (HMAC) and extracts the label
  3. The orchestrator resolves the label to a pool and provisions a runner
  4. The runner registers with GitHub using a JIT token and executes the job

See the Quickstart for full GitHub Actions setup instructions.

GitLab CI

The GitLab adapter acts as a replacement for gitlab-runner. It registers with your GitLab instance as a runner and translates the GitLab runner protocol to iRunner's internal format.

How it works

  1. The GitLab adapter polls GitLab for pending jobs (like a standard gitlab-runner)
  2. When a job is received, it extracts resource requirements from the job's tags
  3. Tags are mapped to iRunner pools (similar to label_mappings)
  4. A runner is provisioned from the matched pool
  5. The adapter provides a GitHub Actions compatibility layer so the runner can execute GitLab CI steps

Configuration

config.yaml
adapters:
  gitlab:
    enabled: true
    url: "https://gitlab.example.com"
    token_secret: "ir-west/gitlab-runner-token"  # Secrets Manager reference
    tag_mappings:
      ir-large: linux-large       # GitLab tag → iRunner pool
      ir-docker: linux-docker
      ir-arm64: linux-arm64
    poll_interval: 5s
    max_concurrent: 20

GitLab CI workflow example

.gitlab-ci.yml
build:
  tags:
    - ir-large              # Resolved to iRunner pool
  script:
    - make build
    - make test

deploy:
  tags:
    - ir-docker
  script:
    - docker build -t myapp .
    - docker push myapp:latest

Harness CI

The Harness adapter integrates with Harness CI's delegate model. The iRunner control plane exposes a local endpoint on port 3000 that the Harness delegate connects to for runner provisioning.

How it works

  1. The Harness delegate runs alongside iRunner (or in the same VPC)
  2. When a Harness CI pipeline stage requires VM infrastructure, the delegate calls iRunner on port 3000
  3. iRunner receives the setup request, provisions a runner from the appropriate pool
  4. Steps are delivered to the runner via the delegate's step execution protocol
  5. On completion, iRunner destroys the runner

Configuration

config.yaml
adapters:
  harness:
    enabled: true
    listen_port: 3000
    pool_mappings:
      # Harness infra definition → iRunner pool
      linux-medium: linux-large
      linux-large: linux-xlarge
      docker: linux-docker

Harness pipeline example

Harness pipeline (VM infrastructure)
stages:
  - stage:
      name: Build
      type: CI
      spec:
        infrastructure:
          type: VM
          spec:
            pool_name: linux-medium    # Mapped to iRunner pool
        execution:
          steps:
            - step:
                type: Run
                spec:
                  command: make build

Shared compute pools

All three platforms use the same underlying pool definitions. A pool named linux-large can serve:

  • GitHub Actions jobs with runs-on: ir-large
  • GitLab CI jobs with tags: [ir-large]
  • Harness CI stages with pool_name: linux-large

This means you manage one set of compute infrastructure regardless of how many CI platforms your organization uses.

Note

Platform adapters are independent — you can enable just GitHub Actions, or any combination of the three. Each adapter is configured separately in the adapters section of config.yaml.
