Observability

iRunner provides full observability into your CI infrastructure through OpenTelemetry traces, Prometheus metrics, CloudWatch integration, and structured JSON logs. Understand job lifecycle, diagnose failures, and track fleet health.

Distributed tracing

Every job produces an OpenTelemetry trace that spans the entire lifecycle from webhook receipt to runner termination. Traces are exported via OTLP to your configured collector.

Trace span structure

Trace spans
job (root span)
├── webhook.receive          # Webhook received and validated
├── webhook.process          # Label lookup and pool resolution
├── runner.provision         # Compute resource creation
│   ├── ec2.run_instance     # (or ecs.run_task / eks.create_pod)
│   └── runner.bootstrap     # Runner agent startup and registration
├── runner.idle              # Waiting for GitHub to assign the job
├── job.execute              # Job execution (GitHub controls this)
└── runner.terminate         # De-register and destroy compute

Span attributes

AttributeDescription
ir.job.idGitHub workflow job ID
ir.job.repoRepository full name
ir.job.workflowWorkflow filename
ir.pool.namePool that served the job
ir.runner.instance_idEC2 instance ID / ECS task ARN / Pod name
ir.runner.spotWhether spot pricing was used
ir.runner.archArchitecture (amd64 / arm64)

Configuration

config.yaml
observability:
  tracing:
    enabled: true
    exporter: otlp
    endpoint: "http://otel-collector.monitoring:4317"
    sampling_rate: 1.0    # 1.0 = trace every job

Prometheus metrics

The control plane exposes a /metrics endpoint in Prometheus exposition format. Key metrics:

Job metrics

Prometheus metrics
# Jobs in queue waiting for a runner
ir_queue_depth{pool}

# Jobs currently executing
ir_jobs_running{pool, instance_type}

# Total jobs completed (counter)
ir_jobs_completed_total{pool, status, spot}

# Job duration histogram
ir_job_duration_seconds_bucket{pool, status}

# Time from webhook to job start (queue + provision time)
ir_job_pickup_duration_seconds_bucket{pool}

# Provisioning duration (time to create compute resource)
ir_provision_duration_seconds_bucket{pool, platform}

Pool metrics

Prometheus metrics
# Current pool utilization
ir_pool_runners_active{pool}
ir_pool_runners_idle{pool}
ir_pool_runners_total{pool}

# Warm pool status
ir_pool_warm_target{pool}
ir_pool_warm_actual{pool}

# Pool capacity (max concurrent runners)
ir_pool_capacity{pool}

Infrastructure metrics

Prometheus metrics
# Webhook processing
ir_webhook_received_total{event_type}
ir_webhook_processing_duration_seconds_bucket{}
ir_webhook_errors_total{error_type}

# API rate limiting (GitHub API)
ir_github_api_requests_total{endpoint}
ir_github_api_rate_remaining{resource}

# Cache service
ir_cache_hit_total{repo}
ir_cache_miss_total{repo}
ir_cache_upload_bytes_total{repo}
ir_cache_download_bytes_total{repo}

CloudWatch integration

Metrics are published to CloudWatch under the iRunner namespace. This enables native AWS alarms, dashboards, and anomaly detection without running a separate Prometheus stack.

config.yaml
observability:
  cloudwatch:
    enabled: true
    namespace: "iRunner"
    region: us-west-2
    publish_interval: 60s

Structured logging

All iRunner components emit structured JSON logs to stdout. Log fields include:

Example log entry
{
  "level": "info",
  "ts": "2024-03-15T10:23:45.123Z",
  "msg": "runner provisioned",
  "component": "orchestrator",
  "job_id": "12345678",
  "repo": "myorg/myrepo",
  "pool": "linux-large",
  "instance_id": "i-0abc123def456",
  "instance_type": "m7i.2xlarge",
  "spot": true,
  "provision_duration_ms": 2340
}

Logs are designed for ingestion into CloudWatch Logs, Datadog, Splunk, or any JSON-capable log aggregator.

Health checks

The control plane exposes health endpoints:

EndpointPurpose
/healthBasic liveness check (ALB target group health)
/readyReadiness check (dependencies connected)
/internal/statusDetailed status including pool states, queue depth, version
Tip

Set up alerts on ir_queue_depth (jobs waiting too long) and ir_pool_warm_actual < ir_pool_warm_target (warm pool below target) to catch issues before they impact developers.
