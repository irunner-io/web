Pools & Labels

Pools define the compute configurations that back your CI jobs. Labels in your workflow's runs-on: field are mapped to pools via label_mappings in config.yaml, giving developers simple labels while platform teams control the underlying infrastructure.

How it works

The relationship between labels and pools is a clean separation of concerns:

  • Developers pick a label in their workflow file (e.g., runs-on: ir-large)
  • Platform teams define what that label resolves to — instance type, disk size, AMI, networking, and more

When a workflow_job webhook arrives, the control plane looks up the requested label in label_mappings, resolves it to a pool definition, and provisions a runner with that configuration.

Pool definition

A pool specifies everything about the compute environment for a runner:

FieldDescriptionExample
nameUnique pool identifierlinux-large
platformCompute backendec2, ecs, eks
instance_typeEC2 instance typem7i.2xlarge
amiAMI ID (EC2 only)ami-0abc123def456
disk_size_gbRoot volume size100
subnet_idsVPC subnets to launch in[subnet-aaa, subnet-bbb]
security_group_idsSecurity groups to attach[sg-123456]
spot_enabledUse spot/preemptible instancestrue
min_warmIdle pre-provisioned runners2

Label mappings

The label_mappings section maps workflow labels to pool names. A single pool can serve multiple labels.

Complete config.yaml example

config.yaml
pools:
  - name: linux-small
    platform: ec2
    instance_type: m7i.large
    ami: ami-0abc123def456
    disk_size_gb: 50
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
    security_group_ids:
      - sg-runners-001
    spot_enabled: false

  - name: linux-large
    platform: ec2
    instance_type: m7i.2xlarge
    ami: ami-0abc123def456
    disk_size_gb: 100
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
    security_group_ids:
      - sg-runners-001
    spot_enabled: true
    min_warm: 2

  - name: fargate-standard
    platform: ecs
    cpu: 2048
    memory: 4096
    image: "123456789.dkr.ecr.us-west-2.amazonaws.com/runner:latest"

label_mappings:
  ir-small: linux-small
  ir-medium: linux-small
  ir-large: linux-large
  ir-ecs: fargate-standard
  ir-cold: linux-large

Using labels in workflows

Once mappings are configured, developers use labels like any other GitHub Actions runner label:

.github/workflows/ci.yml
jobs:
  unit-tests:
    runs-on: ir-small
    steps:
      - uses: actions/checkout@v4
      - run: make test

  integration-tests:
    runs-on: ir-large
    steps:
      - uses: actions/checkout@v4
      - run: make integration-test

  container-build:
    runs-on: ir-ecs
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t myapp .

Multiple labels per job

If a workflow specifies multiple labels (as an array), the control plane matches on the first label that exists in label_mappings:

.github/workflows/ci.yml
jobs:
  build:
    runs-on: [ir-large, ir-cold]  # matches ir-large first
    steps:
      - run: echo "Running on large pool"
Note

If none of the labels in runs-on: match any entry in label_mappings, the job is ignored by iRunner and left for GitHub-hosted runners or other self-hosted runners to pick up.

Best practices

  • Use a consistent prefix (like ir-) to clearly identify iRunner labels
  • Map multiple meaningful labels to the same pool for readability (ir-build, ir-ci can both point to the same pool)
  • Start with a small number of pools and add more as workload patterns emerge
  • Document your label-to-pool mappings for your development team
