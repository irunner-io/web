S3 Cache

iRunner provides a drop-in replacement for actions/cache@v4 that routes cache data directly to S3 in your AWS region. No 10 GB cap, no cross-region latency, and multipart upload for large caches.

Why replace actions/cache?

GitHub's built-in cache has several limitations for self-hosted runners:

LimitationGitHub cacheiRunner S3 cache
Size limit per entry10 GBUnlimited (S3 limits)
Total cache per repo10 GBUnlimited
LocationGitHub's infrastructureYour S3 bucket, your region
LatencyInternet round-tripSame-region S3 (milliseconds)
Large file uploadSingle-part, can timeoutMultipart upload
Retention7 days unusedConfigurable TTL (default 14 days)

Usage

Replace actions/cache@v4 with ir-cache/cache@v1 in your workflow. The API is identical:

.github/workflows/ci.yml
jobs:
  build:
    runs-on: ir-large
    steps:
      - uses: actions/checkout@v4

      - uses: ir-cache/cache@v1
        with:
          path: |
            ~/.npm
            node_modules
          key: npm-${{ runner.os }}-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            npm-${{ runner.os }}-

      - run: npm ci
      - run: npm test

Inputs

InputDescriptionRequired
pathPaths to cache (supports globs and multiline)Yes
keyUnique cache key for this entryYes
restore-keysFallback key prefixes for partial matchesNo

Outputs

OutputDescription
cache-hittrue if exact key match found

How it works

  1. The runner has an IR_CACHE_URL environment variable pointing to the cache service
  2. On restore: the action calls the cache service, which looks up the key in the metadata store and streams the tarball from S3
  3. On save: the action uploads the tarball via multipart upload to S3 through the cache service
  4. Cache entries are keyed by repository, branch scope, and the user-provided key

Multipart upload

For caches larger than 100 MB, the action automatically uses S3 multipart upload:

  • Files are split into 64 MB parts and uploaded in parallel
  • Failed parts are retried individually (no need to restart the entire upload)
  • Supports caches up to 5 TB (S3's maximum object size)

Configuration

The cache service is configured in config.yaml:

config.yaml
cache:
  enabled: true
  bucket: ir-west-cache-123456789012
  region: us-west-2
  ttl_days: 14                    # cache entries older than this are evicted
  max_entry_size_gb: 50           # reject uploads larger than this
  multipart_threshold_mb: 100     # switch to multipart above this size
  multipart_part_size_mb: 64      # size of each upload part

Cache scoping

Cache keys are scoped hierarchically to prevent collisions and enable branch-based isolation:

  • Repository scope: caches are isolated per repository
  • Branch scope: a feature branch can read caches from its base branch but only writes to its own namespace
  • Key matching: exact key first, then restore-keys prefix matching (most recent match wins)

TTL and eviction

Cache entries are automatically evicted based on the ttl_days setting. The eviction is implemented via S3 lifecycle rules applied to the cache bucket. Entries that haven't been accessed within the TTL window are deleted.

Tip

Monitor your cache hit rate via the ir_cache_hit_total and ir_cache_miss_total Prometheus metrics. A low hit rate may indicate your cache keys are too specific.

Migrating from actions/cache

Migration is a one-line change per workflow step:

Before
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: npm-${{ hashFiles('package-lock.json') }}
After
- uses: ir-cache/cache@v1
  with:
    path: ~/.npm
    key: npm-${{ hashFiles('package-lock.json') }}

All input and output parameters are identical. No other workflow changes are needed.
