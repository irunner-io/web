Security Model

iRunner is designed with a minimal attack surface. A single public endpoint (HMAC-verified webhook), default-deny ALB rules, private internal communication, ephemeral runners, and least-privilege IAM roles form the security foundation.

Architecture overview

The security model is built on defense-in-depth principles:

  • Single public surface: Only the webhook endpoint is exposed to the internet
  • HMAC verification: Every webhook payload is verified against the shared secret
  • Default deny: ALB returns 403 for all paths except the webhook
  • Private networking: All internal communication happens over private DNS
  • Ephemeral compute: No persistent attack surface on runner infrastructure

Network security

ALB configuration

The Application Load Balancer has two rules:

PriorityConditionAction
1POST /github/webhookForward to control plane target group
DefaultAll other requestsReturn 403 (fixed response)

This means port scanning, path enumeration, and other reconnaissance techniques return nothing useful to an attacker.

Internal communication

Components communicate via AWS Cloud Map (private DNS within the VPC):

  • Control plane to runners: private subnet, security group restricted
  • Runners to cache service: private DNS (cache.ir.internal)
  • Control plane to DynamoDB/SQS/S3: VPC endpoints (no internet traversal)

Optional WAF

For additional protection, attach AWS WAF to the ALB:

WAF configuration options
# IP allowlisting (restrict to GitHub webhook IPs)
# GitHub publishes their webhook IP ranges at:
# https://api.github.com/meta (hooks field)

# Geographic restrictions
# Block traffic from regions where you don't operate

Webhook verification

Every incoming webhook is verified using HMAC-SHA256:

  1. GitHub signs the payload with your webhook secret
  2. The signature is sent in the X-Hub-Signature-256 header
  3. iRunner recomputes the HMAC and compares (constant-time comparison)
  4. Invalid signatures are rejected with 401 — no further processing occurs
Note

The webhook secret is stored in AWS Secrets Manager and never written to disk or logged. It is loaded into memory at control plane startup.

Secrets management

All sensitive values are stored in AWS Secrets Manager:

  • GitHub App private key: Used to generate installation tokens
  • Webhook secret: Used for HMAC verification
  • License key: Validates entitlements

Secrets are never written to environment variables in plaintext (except within the control plane process memory), never logged, and never stored on disk.

Ephemeral runner security

The ephemeral model provides strong isolation guarantees:

  • Runners use JIT (just-in-time) registration tokens valid for a single use
  • No long-lived runner tokens exist in the system
  • Each runner has a unique, short-lived GitHub token scoped to the specific job
  • The compute resource is destroyed immediately after job completion
  • No persistent filesystem, no surviving processes, no network presence after teardown

Supply chain security

  • iRunner binaries are signed and distributed via S3 with integrity verification
  • Base AMIs and container images are built from minimal, audited sources
  • CloudFormation templates are versioned and published with checksums
  • No third-party dependencies at runtime (single static Go binary)

Security hardening checklist

  • Restrict ALB security group inbound to GitHub webhook IP ranges only
  • Enable AWS WAF with rate limiting on the ALB
  • Use VPC endpoints for all AWS service access (no internet required for internal traffic)
  • Enable CloudTrail for IAM audit logging
  • Rotate the webhook secret periodically
  • Use separate AWS accounts for production and development iRunner deployments
  • Enable GuardDuty for anomaly detection on the iRunner VPC
