Spot Instances

Spot instances let you run CI jobs at up to 90% discount compared to on-demand pricing. iRunner supports EC2 Spot and Fargate Spot with automatic fallback to on-demand when spot capacity is unavailable.

How it works

When spot_enabled: true is set on a pool:

  1. The control plane requests spot capacity first using the capacity-optimized allocation strategy
  2. If spot capacity is available, the runner launches at the spot price
  3. If spot capacity is unavailable (InsufficientInstanceCapacity), the control plane automatically falls back to on-demand
  4. The fallback is transparent — the job runs regardless of which pricing model was used

Configuration

config.yaml
pools:
  - name: linux-spot
    platform: ec2
    instance_type: m7i.2xlarge
    ami: ami-0abc123def456
    disk_size_gb: 100
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
      - subnet-ccc333    # multiple AZs increase spot availability
    security_group_ids:
      - sg-runners-001
    spot_enabled: true
    spot_fallback: true   # fall back to on-demand (default: true)

  - name: fargate-spot
    platform: ecs
    cpu: 4096
    memory: 8192
    image: "123456789.dkr.ecr.us-west-2.amazonaws.com/runner:latest"
    spot_enabled: true    # uses Fargate Spot capacity provider

label_mappings:
  ir-spot: linux-spot
  ir-ecs-spot: fargate-spot

Allocation strategy

iRunner uses the capacity-optimized allocation strategy for EC2 Spot. This strategy launches instances from the pool with the most available capacity, reducing the likelihood of interruption compared to lowest-price strategies.

For Fargate Spot, the allocation is managed by AWS — the task is scheduled on available Fargate Spot capacity automatically.

Spot interruption handling

AWS provides a 2-minute warning before terminating a Spot instance. When this happens:

  • The running job will fail (the runner is terminated mid-execution)
  • GitHub Actions marks the job as failed
  • The job can be retried (manually or via workflow retry configuration)
Note

Spot instances can be interrupted at any time. Only use spot for stateless, idempotent jobs that can be safely retried. Do not use spot for jobs that modify external state without idempotency guarantees (e.g., deploying to production without rollback capability).

Best practices

  • Use multiple AZs: Spread subnet_ids across 3+ availability zones to maximize spot availability
  • Use for CI, not CD: Spot is ideal for test suites, linting, and builds — not production deployments
  • Combine with on-demand pools: Use spot for bulk testing and on-demand for critical-path jobs
  • Keep jobs short: Shorter jobs are less likely to be interrupted mid-execution
  • Enable fallback: Keep spot_fallback: true (the default) so jobs still run when spot is unavailable

Separate labels for spot vs on-demand

A common pattern is to expose both options via different labels:

config.yaml
pools:
  - name: linux-ondemand
    platform: ec2
    instance_type: m7i.2xlarge
    spot_enabled: false
    # ... other fields

  - name: linux-spot
    platform: ec2
    instance_type: m7i.2xlarge
    spot_enabled: true
    # ... other fields

label_mappings:
  ir-large: linux-ondemand     # guaranteed capacity
  ir-large-spot: linux-spot    # cheaper but interruptible
