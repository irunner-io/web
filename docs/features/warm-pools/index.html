Warm Pools

Warm pools keep pre-provisioned runners idle and ready to accept jobs instantly. By maintaining a minimum number of warm runners, you eliminate cold-start latency at the cost of keeping idle compute running.

The cold-start problem

Without warm pools, every job triggers fresh provisioning:

  • EC2: 30-90 seconds to launch an instance, boot the OS, and register the runner
  • ECS Fargate: 15-45 seconds to pull the image and start the task
  • EKS: 10-30 seconds to schedule and start a pod

For teams that need sub-second job pickup (e.g., latency-sensitive CI/CD pipelines or developer-facing checks), warm pools eliminate this wait entirely.

How it works

  1. You set min_warm: N on a pool
  2. The housekeeping loop continuously monitors the number of idle runners for that pool
  3. If the idle count drops below min_warm (because a runner was assigned a job), the housekeeping loop launches replacement runners
  4. Warm runners are pre-registered with GitHub and waiting in the idle state
  5. When a job arrives, an idle runner is immediately assigned — no provisioning delay

Configuration

config.yaml
pools:
  - name: linux-fast
    platform: ec2
    instance_type: m7i.xlarge
    ami: ami-0abc123def456
    disk_size_gb: 100
    subnet_ids:
      - subnet-aaa111
      - subnet-bbb222
    security_group_ids:
      - sg-runners-001
    spot_enabled: false
    min_warm: 2              # always keep 2 idle runners ready

  - name: fargate-warm
    platform: ecs
    cpu: 4096
    memory: 8192
    image: "123456789.dkr.ecr.us-west-2.amazonaws.com/runner:latest"
    min_warm: 3              # keep 3 tasks ready for instant assignment

label_mappings:
  ir-fast: linux-fast
  ir-ecs-warm: fargate-warm

Housekeeping reconciliation

The control plane's housekeeping loop runs on a configurable interval (default: 30 seconds) and performs the following for each pool with min_warm set:

  1. Counts the current number of idle (unassigned) runners for the pool
  2. If idle count < min_warm, launches min_warm - idle_count new runners
  3. If idle count > min_warm (unlikely, but possible after scaling events), excess runners are left to timeout naturally
Note

Warm runners still have a maximum idle timeout (default: 30 minutes). If a warm runner sits idle for too long without receiving a job, it is terminated and immediately replaced by the housekeeping loop. This ensures the fleet stays fresh.

Cost tradeoff

Warm pools trade cost for latency. Consider the math:

Instance typeHourly costmin_warm=2 monthly cost
m7i.large (2 vCPU, 8 GB)$0.101~$146/mo
m7i.xlarge (4 vCPU, 16 GB)$0.202~$291/mo
m7i.2xlarge (8 vCPU, 32 GB)$0.403~$581/mo

This is the worst-case idle cost — in practice, warm runners are frequently used for jobs, so the effective idle cost is lower during business hours.

When to use warm pools

  • Use warm pools when: developer experience matters, PR checks need fast feedback, or you have a steady baseline of jobs throughout the day
  • Skip warm pools when: jobs are infrequent, cost optimization is the top priority, or cold-start latency is acceptable for your workflows

Combining with spot instances

You can combine min_warm with spot_enabled: true to reduce the cost of warm runners. However, be aware that spot interruptions may temporarily reduce your warm pool below the target count until housekeeping replaces the interrupted instances.
