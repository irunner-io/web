iRunner Documentation

iRunner provides intelligent, self-hosted GitHub Actions runners that execute inside your AWS account. You keep the full GitHub Actions ecosystem — workflows, marketplace actions, triggers — while running builds on infrastructure you control.

How iRunner works

iRunner sits between GitHub and your AWS infrastructure:

  1. GitHub sends a workflow_job webhook when a job is queued
  2. The iRunner control plane receives the webhook and matches the runs-on label to a configured pool
  3. A runner is provisioned (EC2 instance, ECS task, or EKS pod) in your VPC
  4. The runner registers with GitHub, executes the job, and is destroyed

The only public endpoint is a single webhook URL. All other communication happens over internal AWS networking.

Architecture

The control plane runs as an ECS service behind an Application Load Balancer. It manages:

  • Webhook ingestion — receives and validates GitHub webhooks
  • Pool management — maintains warm capacity and provisions on demand
  • Job routing — matches runs-on labels to compute pools
  • Cache service — S3-backed cache compatible with actions/cache
  • Lifecycle management — terminates runners after job completion

Compute backends

iRunner supports three compute backends, each suited to different workload types:

Backend Best for Cold start Isolation
EC2 Docker builds, GPU, large disk, custom kernels 45–90s (or instant with warm pools) Full VM
ECS Fargate Fast, lightweight jobs, no AMI management 15–30s Container
EKS Teams with existing Kubernetes clusters 10–20s (on warm nodes) Pod
