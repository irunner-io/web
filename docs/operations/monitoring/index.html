Monitoring

iRunner exposes structured logs, Prometheus metrics, and OpenTelemetry traces to give you full visibility into the control plane and runner lifecycle.

Structured logs

The control plane emits structured JSON logs to CloudWatch Logs:

Log groupContents
/ecs/{prefix}-control-planeControl plane application logs

Each log line includes fields for level, msg, job_id, pool, repo, and duration_ms for easy filtering.

Prometheus metrics

Metrics are exposed at :8080/metrics in Prometheus exposition format. Scrape this endpoint from your monitoring stack.

MetricTypeDescription
ir_jobs_totalCounterTotal jobs processed (labels: pool, status)
ir_jobs_activeGaugeCurrently running jobs
ir_pool_capacityGaugeMaximum runners per pool
ir_pool_warm_countGaugeWarm (idle) runners ready in each pool
ir_job_duration_secondsHistogramEnd-to-end job duration
ir_cache_hits_totalCounterCache hit/miss counts (labels: pool, result)
ir_cache_size_bytesGaugeTotal cache storage used per pool

OpenTelemetry traces

iRunner produces distributed traces covering the full job lifecycle. Each job generates a trace with the following spans:

  • webhook_received — GitHub delivers the workflow_job event
  • pool_matched — label routing selects a pool
  • runner_provisioned — compute resource launched (EC2/ECS/EKS)
  • job_started — runner registers with GitHub and begins execution
  • job_completed — job finishes (success or failure)
  • runner_terminated — ephemeral runner cleaned up

Configure the OTLP exporter endpoint in config.yaml to send traces to your collector (Jaeger, Tempo, etc.).

CloudWatch dashboards

Monitor ECS service health through the AWS Console or create custom dashboards for:

  • ECS task count (running vs desired)
  • CPU and memory utilization of control plane tasks
  • Task restart count and failure reasons

RequestCountWebhook delivery rate
HTTPCode_Target_5XX_CountControl plane errors (should be near zero)

DynamoDB metrics

MetricWhat to watch
ConsumedReadCapacityUnitsRead throughput against provisioned capacity
ConsumedWriteCapacityUnitsWrite throughput against provisioned capacity
ThrottledRequestsMust remain at zero; scale capacity if non-zero

SQS metrics

MetricWhat to watch
ApproximateNumberOfMessagesVisibleQueue depth — jobs waiting to be dispatched
ApproximateAgeOfOldestMessageHow long the oldest job has been waiting

Recommended alerts

ConditionThresholdAction
SQS queue depth> 10 for 5 minutesInvestigate pool capacity or provisioning failures
ALB 5xx rate> 1% of requestsCheck control plane logs for errors
ECS task failuresAny task exit code != 0Review task stopped reason in ECS console
DynamoDB throttlesAny ThrottledRequests > 0Increase table capacity or switch to on-demand
Tip

Use CloudWatch Logs Insights to correlate job IDs across log streams. Query with filter job_id = "abc123" to trace a single job through the entire system.
