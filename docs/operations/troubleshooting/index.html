Troubleshooting

Common issues, their root causes, and how to resolve them. When in doubt, enable debug logging and check the control plane health endpoint.

Common issues

SymptomCauseFix
Job stuck in "Queued" Webhook not reaching control plane Verify ALB security group allows inbound 443 from GitHub IPs. Check webhook secret matches. Confirm GitHub App subscription to workflow_job event.
Runner fails to register GitHub App permissions or expired token Ensure the App has Self-hosted runners: Read & Write on the org. Verify the private key in Secrets Manager is current. Check control plane logs for token refresh errors.
Cache returns 403 Forbidden Wrong config path or missing IAM permissions Confirm cache.bucket in config.yaml matches the actual S3 bucket name. Verify the runner instance profile has s3:GetObject and s3:PutObject on the cache bucket ARN.
EC2 instance not launching Subnet capacity, AMI not found, or missing instance profile Check subnet has available IPs. Verify AMI ID exists in the configured region. Confirm the instance profile ARN in pool config is correct and the role is assumable.
ECS task fails to start Image not in ECR, task role permissions, or resource limits Verify the image exists in ECR with the expected tag. Check the task execution role has ecr:GetAuthorizationToken and ecr:BatchGetImage. Review CPU/memory limits in task definition.
Control plane crashloop config.yaml syntax error or inaccessible secrets Check CloudWatch logs for the specific parse error. Validate config.yaml with ir validate-config. Ensure Secrets Manager ARNs in the task definition are accessible by the task role.
Spot interruption causes job failure AWS reclaims spot instance mid-execution Configure an on-demand fallback pool. Set spot.fallback: on-demand in pool config. Use warm pools to reduce cold-start impact of replacement instances.

Debug mode

Enable verbose logging by setting the IR_LOG_LEVEL environment variable on the ECS task definition:

Terminal
# Update the task definition environment variables
# In deploy/task-definition.json, add:
{
  "name": "IR_LOG_LEVEL",
  "value": "debug"
}

# Re-register and deploy
aws ecs register-task-definition --cli-input-json file://deploy/task-definition.json
aws ecs update-service \
  --cluster {prefix}-cluster \
  --service {prefix}-control-plane \
  --task-definition ir-control-plane \
  --force-new-deployment

Health check endpoint

The control plane exposes a /health endpoint that returns component status:

Terminal
curl -s http://control-plane.{prefix}.internal:8080/health | jq .
{
  "status": "healthy",
  "components": {
    "dynamodb": "ok",
    "sqs": "ok",
    "s3": "ok",
    "github": "ok"
  },
  "uptime_seconds": 84321
}

Log collection with CloudWatch Logs Insights

Use these queries to investigate issues:

Find errors in the last hour
fields @timestamp, @message
| filter level = "error"
| sort @timestamp desc
| limit 50
Trace a specific job
fields @timestamp, msg, level
| filter job_id = "12345678"
| sort @timestamp asc
Find slow provisioning (> 60s)
fields @timestamp, job_id, pool, duration_ms
| filter msg = "runner_provisioned" and duration_ms > 60000
| sort duration_ms desc
| limit 20
Count failures by pool
fields pool
| filter level = "error"
| stats count(*) as errors by pool
| sort errors desc
Tip

The internal /internal/status endpoint provides a richer view including pool states, queue depths, and active job counts. Access it from within the VPC via SSM Session Manager or a bastion host.
