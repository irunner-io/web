Upgrades

Upgrading iRunner is a zero-downtime operation. Build a new image, push it to ECR, and update the ECS service. The rolling deployment ensures continuous availability.

Step 1: Build the new image

Tag the image with a version identifier for traceability:

Terminal
VERSION=$(git describe --tags --always)

docker build -t ir-control-plane:${VERSION} .
docker tag ir-control-plane:${VERSION} \
  {account-id}.dkr.ecr.us-west-2.amazonaws.com/ir-control-plane:${VERSION}

Step 2: Push to ECR

Terminal
aws ecr get-login-password --region us-west-2 | \
  docker login --username AWS --password-stdin {account-id}.dkr.ecr.us-west-2.amazonaws.com

docker push {account-id}.dkr.ecr.us-west-2.amazonaws.com/ir-control-plane:${VERSION}

Step 3: Register new task definition

Create a new task definition revision pointing to the new image:

Terminal
# Update the image in the task definition JSON
sed -i "s|ir-control-plane:.*|ir-control-plane:${VERSION}\"|" deploy/task-definition.json

# Register the new revision
aws ecs register-task-definition \
  --cli-input-json file://deploy/task-definition.json

Step 4: Update ECS service

The ECS service performs a rolling deployment with minimum healthy percent set to 100%, ensuring no downtime:

Terminal
aws ecs update-service \
  --cluster {prefix}-cluster \
  --service {prefix}-control-plane \
  --task-definition ir-control-plane \
  --force-new-deployment

Monitor the deployment:

Terminal
aws ecs wait services-stable \
  --cluster {prefix}-cluster \
  --services {prefix}-control-plane

Configuration changes

If your upgrade requires config changes, update the config in S3 and restart the ECS tasks:

Terminal
# Upload updated config
aws s3 cp config.yaml s3://{prefix}-releases-{account-id}/config/config.yaml

# Force new deployment to pick up the new config
aws ecs update-service \
  --cluster {prefix}-cluster \
  --service {prefix}-control-plane \
  --force-new-deployment

Stack updates

For infrastructure-level changes (new subnets, modified IAM policies, etc.), update the CloudFormation stack:

Terminal
aws cloudformation update-stack \
  --stack-name ir-controlplane \
  --template-url https://ir-releases.s3.amazonaws.com/cloudformation/latest.yaml \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --use-previous-template \
  --parameters \
    ParameterKey=Prefix,UsePreviousValue=true \
    ParameterKey=VpcCidr,UsePreviousValue=true \
    ParameterKey=GitHubAppId,UsePreviousValue=true \
    ParameterKey=GitHubInstallationId,UsePreviousValue=true

Zero-downtime guarantees

  • ECS rolling update launches new tasks before draining old ones
  • ALB health checks (/health endpoint) gate traffic to new tasks
  • Minimum healthy percent of 100% ensures full capacity during deployment
  • In-flight jobs continue on existing runners unaffected by control plane restarts

Rollback

To roll back, revert to the previous task definition revision:

Terminal
# List recent task definition revisions
aws ecs list-task-definitions \
  --family-prefix ir-control-plane \
  --sort DESC --max-items 5

# Update service to previous revision
aws ecs update-service \
  --cluster {prefix}-cluster \
  --service {prefix}-control-plane \
  --task-definition ir-control-plane:{previous-revision} \
  --force-new-deployment
Tip

Keep at least 3 previous task definition revisions available for quick rollback. ECS retains inactive revisions indefinitely by default.
