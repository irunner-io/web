The hard parts, handled

Built for enterprise runner operations, not weekend runner scripts.

Self-hosting the runner binary is easy. Operating secure, elastic, observable CI capacity across many repositories is the real platform problem.

01

Private deployment boundary

Control plane in private AWS networking, API Gateway for HTTPS ingress, and workers launched inside your account.

02

Ephemeral workers by default

Each job gets clean compute and a short-lived GitHub runner registration, then the worker disappears.

03

S3-backed cache and artifacts

Keep dependency caches and build outputs in-region, close to runners, without changing standard Actions steps.

04

Multi-backend placement

Use EC2 for VM isolation, ECS Fargate for serverless jobs, and EKS when Kubernetes is the right execution layer.

05

Enterprise audit posture

Runner lifecycle, cache writes, queue events, and AWS resources are designed around traceable ownership.

06

Platform-owned defaults

Ship labels such as ir-linux-medium so app teams do not size pools or learn new CI primitives.

Platform

One GitHub App. One control plane. Multiple execution backends.

1

GitHub queues a job

A workflow_job.queued webhook reaches iRunner.

2

iRunner chooses capacity

Labels map to EC2, ECS, or EKS capacity with just-in-time runner registration.

3

The worker runs the job

The runner uses GitHub Actions normally while cache and artifact traffic stays close to AWS.

4

The worker is cleaned up

State returns to iRunner and AWS services; the job compute does not become a long-lived runner.
